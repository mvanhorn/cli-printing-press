# Auto-mirror workflow trigger gaps: PRs check committed state, but mirror workflows fire only on push to main

## Context

> **Cross-repo note.** The mirror invariant ("`cli-skills/pp-<api>/SKILL.md` is a byte-identical copy of `library/<cat>/<api>/SKILL.md`") and its regeneration workflow live in [`mvanhorn/printing-press-library`](https://github.com/mvanhorn/printing-press-library). The generator that produces the source-tree content is in `cli-printing-press` (this repo). This learning is filed here because the friction surfaces when our generator output flows downstream — anyone shipping a template change here needs to know what the downstream PR loop expects.

The published library has two trees that must stay in sync:

- `library/<cat>/<api>/SKILL.md` — canonical content, generated by `cli-printing-press` and committed per CLI
- `cli-skills/pp-<api>/SKILL.md` — flat-namespace mirror of the above, used by `npx skills add` and Hermes / OpenClaw install paths

The mirror is produced by `tools/generate-skills/main.go`, run automatically by `.github/workflows/generate-skills.yml`. After PR #266 simplified the generator to a verbatim mirror (no synthesis fallback), the workflow's job became: read each `library/<cat>/<api>/SKILL.md`, copy verbatim to `cli-skills/pp-<api>/SKILL.md`.

The workflow fires on push to main:

```yaml
# printing-press-library/.github/workflows/generate-skills.yml
on:
  push:
    branches: [main]
    paths:
      - 'registry.json'
      - 'library/**/SKILL.md'    # added later — see below
      - 'library/**/.printing-press.json'
      - 'tools/generate-skills/main.go'
```

Separately, a `cli-skills mirror parity` check runs on every PR to verify the committed `cli-skills/` matches what the generator would produce from the current `library/`. This catches drift.

When the Hermes / OpenClaw frontmatter alignment work submitted PR #267 (a one-shot sweep that mutated 49 `library/<cat>/<api>/SKILL.md` files), the parity check fired across all 49 mirrored entries:

```
##[error]cli-skills/ is out of sync with what tools/generate-skills produces from library/<cat>/<slug>/SKILL.md.
##[error]Run `go run ./tools/generate-skills/main.go` locally and commit the result.

Out-of-sync entries:
 M cli-skills/pp-agent-capture/SKILL.md
 M cli-skills/pp-ahrefs/SKILL.md
 ... (47 more)
```

The PR mutated the source tree. The mirror check verified the committed mirror against the new source. The auto-mirror workflow that would normally fix this only runs on push to main — it doesn't run on PRs. So the PR author has to manually run `go run ./tools/generate-skills/main.go` and commit the regenerated `cli-skills/` to satisfy the parity check.

This isn't broken — it's deliberate. The workflow re-running on PRs would create commit churn (every push triggers a mirror commit). The parity check on PRs is the right gate; the workflow on push is the right auto-fixer. But the relationship between the two isn't obvious to a PR author seeing the parity failure for the first time.

## Guidance

When changing source-tree content that has a mirror tree, run the mirror generator locally before committing. If the mirror generator's workflow only runs on push to main, the committed-state assumption that PR-side parity checks make is your responsibility to satisfy.

For the printing-press-library case specifically, the workflow:

```bash
# 1. Make changes to library/<cat>/<api>/SKILL.md (or README.md)
# 2. Regenerate the mirror BEFORE committing
go run ./tools/generate-skills/main.go

# 3. Commit both the source change AND the regenerated cli-skills/
git add library/ cli-skills/
git commit -m "..."
```

A trigger fix that lets the PR-side workflow regenerate (rather than just verify) would resolve the friction at the cost of introducing per-PR commit churn. We chose the verify-only-on-PR shape on purpose — it forces the author to think about the mirror as part of the change, rather than letting CI silently fix it.

For the upstream (this repo) side, the lesson is: when shipping a template change that affects the generator output that downstream consumes, the downstream PR loop expects the regenerated mirror to be committed alongside. Document the cross-repo dependency in the upstream commit message so future contributors regenerating downstream content aren't surprised.

## Why This Matters

Three failure modes the local-regen-before-commit workflow prevents:

1. **PR-loop tax.** Without local regen, the PR fails parity, the author runs the regen, force-pushes, CI re-runs, parity passes. That's three CI cycles per oversight, when one would do.
2. **Surprised author hypothesis.** A PR author who hasn't worked in the downstream repo before may not know the mirror exists. The parity-check failure message says "Run `go run ./tools/generate-skills/main.go`" — clear enough — but only if you understand what `tools/generate-skills` does, why the mirror exists at all, and that the workflow in the same repo doesn't run for you on PRs.
3. **Composite-PR confusion.** If the source change AND the mirror change are in the same PR (the right shape), reviewers see one logical change. If they land in two PRs (one for source, one for mirror catch-up), reviewers have to reconstruct the relationship. Land them together.

The trigger expansion in the workflow itself (`library/**/SKILL.md` and `library/**/README.md` added to the `paths:` list as part of the Hermes-alignment work) is what makes this loop work going forward — once the source change merges to main, the workflow runs and mirror updates. Before that addition, mirror updates required someone to push a *different* file too (registry.json, a manifest), which silently masked drift.

## When to Apply

- Working on a PR to a repository with a source/mirror invariant
- The mirror regeneration runs in CI but only on a non-PR trigger (push to main, scheduled, manual dispatch)
- A PR-side parity / verify check exists that compares committed state to "what the generator would produce"
- You're changing source-tree content that the generator transforms

The corollary applies upstream: if you maintain a tool that produces output a downstream repo mirrors, document the cross-repo workflow dependency. The generator output's shape changes have implications beyond the upstream's own CI.

## Examples

### Anti-pattern — change source, push, hit parity failure, fix forward

```bash
# In printing-press-library, oblivious to the mirror
$ vim library/payments/mercury/SKILL.md   # edit content
$ git commit -am "fix(skills): tweak mercury skill"
$ git push  # opens PR

# CI runs:
#   ✓ verify-skill — mercury SKILL.md is fine
#   ✗ Verify (cli-skills mirror parity) — cli-skills/pp-mercury/SKILL.md is stale
#
# Author now needs to:
$ go run ./tools/generate-skills/main.go
$ git add cli-skills/
$ git commit -m "chore(skills): regenerate cli-skills"
$ git push  # CI re-runs
```

Three commits, two CI cycles, for one logical change.

### Pattern — regen locally, commit together

```bash
$ vim library/payments/mercury/SKILL.md
$ go run ./tools/generate-skills/main.go
$ git add library/ cli-skills/
$ git commit -m "fix(skills): tweak mercury skill"
$ git push  # CI passes on first run
```

One commit, one CI cycle.

### What this PR did to make this case visible

PR #267 adjusted the workflow trigger:

```diff
 on:
   push:
     branches: [main]
     paths:
       - 'registry.json'
       - 'library/**/.printing-press.json'
+      - 'library/**/SKILL.md'
+      - 'library/**/README.md'
       - 'tools/generate-skills/main.go'
```

After merge, edits to `library/**/SKILL.md` or `library/**/README.md` on main trigger an auto-mirror commit. Before that addition, hand-edits to library SKILL.md content silently left `cli-skills/` stale until a *different* file change triggered the workflow — exactly the drift class the parity check is supposed to prevent.
